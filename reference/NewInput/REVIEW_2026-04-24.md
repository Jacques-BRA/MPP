## 1. Executive Summary

- **Overall coverage: ~90%.** Every distinct capability on the legacy MES screen map has a documented home in our FDS or a tracked Open Item. The Flexware storyboards were authored in 2012 and describe a superset of what MPP now uses; much of what appears on the screen map (Serialized Item Search as a standalone top-level capability, Material Classes as a first-class entity, the `IdentifierFormat` counter) was already picked up by our 2026-04-22 screenshot review that generated OI-15 through OI-30. This review adds no fundamentally new capabilities; it confirms coverage and surfaces a small number of reinforcement recommendations.
- **Top gap confirmed: Material Allocation screen family.** Flexware has a top-level *Create Menu → Create Container / Create Lot / Create Serialized Item*, a full *Material Allocation* subsystem (`MaterialAllocationMenuView` → `MaterialAllocationView` → `MaterialAllocationCreateView` → `MaterialAllocationUpdateView` → `BomComponentDetailsView`), and a *Current Material Allocation* pane. Our FDS covers the **data** for allocation (OI-18 added Min/Max/Default/Consumption Point on `Parts.ItemLocation` in v1.8) but does not document a **dedicated Material Allocation operator screen** — allocation is currently implicit in Assembly material-verification (FDS-06-011). Recommend adding an explicit §5.11 or §6.6a for the Allocate Material workflow before Arc 2 Phase 6 (Assembly) kicks off. This couples to **UJ-09** (Material verification at assembly — still open).
- **Design confirmations.** Our 2026-04-22 decision to **not reproduce** the Flexware `*DashboardConfiguration` / `UserInterfaceScript` family (Plan v0.2 §H/§I) is validated — the storyboards confirm the legacy pattern is five per-screen configuration tables (`LotCreateDashboardConfiguration`, `LotTrackingDashboardConfiguration`, `WorkOrderDashboardConfiguration`, plus a junction `WorkstationDashboardConfiguration`) driving a single generic view. Every one of those variability surfaces now maps either to a purpose-built Perspective view or to a single LocationAttribute (DefaultScreen, TerminalMode, TrackingMode, LinesideLimit). We are not losing functionality by dropping the pattern.
- **IP addressing: 47 terminals confirmed across two lines.** The spreadsheet confirms the 5G0 Fronts / 5G0 Rears / AP4 assembly-serialized + AP4 M1–M3 machining / AP4 C1 Casting / AP4 T1 Tumbling topology we modelled in §2.4 Location seed data. Each *MES Interface* terminal has a paired *Out MES Printer* (a Zebra) plus in some cases a *Vorne Production Display Board* and a *Machine Integration Panel* — our §2.5 Terminals model with filtered-by-LocationAttribute MIP + Printer lookups covers this **provided** we land the `Workstation.DefaultScreen` / TerminalMode attributes at Arc 2 Phase 1. One-time nit: two DHCP rows for AP4-A2 machines both claim the hostname `AP4-A2-000` — legacy typo; not our problem.
- **Top risk: `IdentifierSequence` cutover (OI-31).** OI-31's open questions (format carry-forward, reset policy, rollover policy) still matter. ~~The storyboards screen map surfaces `IdentifierFormat` as a central entity driving `WorkOrder.IdentifierFormatID` and `ProductionOrder.IdentifierFormatID` — suggesting additional counters.~~ **CORRECTION (2026-04-24):** Jacques confirmed the `IdentifierFormat` export he provided to the session (Lot + SerializedItem, two rows) IS the complete live list. The Flexware FK references from other tables are schema placeholders with no populated format rows. No additional counters in scope.

### 3.2 Observations vs our model

- **Terminal count confirmed: 22 MES-interface terminals** (DHCP sheet row count minus the blank at end). Our §2.4 Seed Data reference to "~230 machines" covers equipment; terminals are a subset. This matches the scale we assumed in Phase 1 seed data planning.
- **Every serialized assembly area has a MIP.** 5G0-A1, 5G0-A2, AP4-A1, AP4-A2 all carry a Mitsubishi PLC + dedicated Machine Integration Panel. No MIP on machining, tumbling, or casting areas — consistent with FDS-10-002 (MIP is serialized-line-only).
- **Every machining area has a dedicated "Out MES Printer"** — a Zebra at each OUT station for LTT reprint (FDS-05-020 `ReprintDamaged`) and presumably sublot label printing (FDS-05-024 Machining sub-LOT split).
- **AP4-M2 has three input terminals** (M2-100 for CH EX1/EX2 + RH5, M2-101 for Fuel Pump) plus an F1 Fuel Pump Printer. Our Location hierarchy models this as M2 being a Line with multiple Cells under it; the F1 designation ("F1 area") suggests a sub-area grouping for Fuel Pump not captured in our Area / Line / Cell tree. **Not a gap** — absorbed by the Cell-tier `LocationAttribute` and category tagging — but flag for Phase 0 Workstation-category discovery (OI-29).
- **Vorne Production Display Boards appear on most cells** — these are public-facing count / downtime displays, not data-collection devices. Out of MES scope for MVP (they consume data we emit via OPC tags; TagHistorian covers this per FDS-01-007).
- **5G0 M-series has only a Vorne board**, no MES terminal. Legacy assumption: 5G0 machining uses the 5G0-M1 terminals shared with the other shift. Reality 2026: unclear. Flag for Phase 0.
- **Hostname convention `{Line}-{Cell}-{NNN}`** — `-100` = In, `-000` = Out, `-101` = secondary In. Useful pattern for terminal `DefaultScreen` seeding at Phase 0 (B11 routing).
- **Static IP range 172.17.20.60–85** — PLCs + MIPs + Vornes + Printers. 26 devices in this block.

## 4. New Gaps Identified

### Gap 1 — Material Allocation Operator Screen (NEW OI-32, medium priority)

Flexware has a dedicated **Material Allocation** capability separate from inspection / material-verification at production time:

- `MaterialAllocationMenuView` (entry tile under Track)
- `MaterialAllocationView` — grid of current allocations at a workstation
- `MaterialAllocationCreateView` — allocate a LOT to a consumption point
- `MaterialAllocationUpdateView` — edit or release an existing allocation
- `BomComponentDetailsView` — drill-in to a specific BOM line

The legacy `Workstation` table (`Flexware_MES_DashboardConfiguration.png`) carries a `MaterialAllocationRequired` BIT per workstation, and the Workstation table in `Flexware_MES_Location_Model.png` carries `MaterialAllocationRequired` — i.e., whether an operator *must* have an explicit allocation in place before the PLC hand-shake can consume parts.

**Our coverage today:** FDS-06-011 covers material **verification** (scan LTT, BOM-check, reject mismatched). OI-18 adds min/max/default/consumption-point metadata on `Parts.ItemLocation` (data model). FDS-06-020 captures the actual `ConsumptionEvent` when PLC cycles fire. **What is missing: a pre-PLC, operator-driven "stage material at this cell" screen and the corresponding `MaterialAllocation` lifecycle table.**

The critical question is whether MPP relies on the allocation step as a discrete pre-production action (e.g., a changeover person pre-positioning N baskets of 59B-fronts at AP4-A2 before first cycle), or whether the live Assembly flow scans-and-consumes in one step (which is what our FDS-06-011 currently assumes).

**Proposed direction:**

- **Option A (recommended):** Open **OI-32 — Material Allocation operator screen**. Phase-0 discovery question: does MPP staff pre-allocate material at assembly stations, or is allocation implicit in first scan? If yes, add a `Workorder.MaterialAllocation` table (Cell, Item, AllocatedQuantity, ConsumedQuantity, AppUserId, AllocatedAt, ReleasedAt) plus a Perspective screen. Phase 6 Assembly work then gates on "allocation exists for this Cell + Item" before the first `ConsumptionEvent` fires, mirroring Flexware's `Workstation.MaterialAllocationRequired` BIT.
- **Option B:** Confirm MPP does not use the step and close the gap as "intentionally not reproduced." Low-risk.

Impact: Phase 6 Assembly work (Arc 2 Plan Phase 6 per v0.2) cannot start design until resolved.

### Gap 2 — Material Classes (NEW OI-32b, low priority, discovery)

Flexware has a `Material` → `MaterialClassID` FK driving a Material Classes tile in Configuration. Our data model has `Parts.ItemType` (Cast / Machined / Assembled / Received) which is coarser. MaterialClass in Flexware looks like it could hold Honda-customer-specific categorisation (per the `Customer` column visible on `WorkOrder`).

**Proposed direction:** Open **OI-32b — Material Classes** as discovery-only. Phase-0 question: does MPP use MaterialClass today? What does the class list look like? If it's Honda-customer groups, `Parts.Item.CustomerCode` (new column) probably covers it. If it's a finer slice of ItemType, `Parts.ItemSubType` or revive the discarded `MaterialClass` table.

### Gap 3 — ~~`IdentifierSequence` additional counters (extend OI-31)~~ — CORRECTED 2026-04-24

**This gap was based on an incorrect inference from the Flexware ERD and has been withdrawn.** The storyboards ERD (`Flexware_MES_OrderDiagram.png`) shows `IdentifierFormat` FKs from `ProductionOrder` and `WorkOrder`, which looked like additional live counters. Jacques confirmed the `IdentifierFormat` export he provided (Lot + SerializedItem, 2 rows) is the complete list — the other FK references are schema placeholders without populated rows. OI-31 scope of two counters is correct as-delivered. No follow-up action.

### Gap 4 — Label Template Configuration Screen

Flexware has a `LabelTemplate` table with ZPL-template content stored in-DB + a `LabelPrinterLabelTemplate` junction assigning templates to printers per WorkCell (see `Flexware_MES_Printing_Model.png`). Our FDS assumes ZPL templates live on the filesystem or in Ignition resource management.

**Risk:** Low. ZPL templates change infrequently (once at cutover + occasional Honda label-format revisions). Filesystem / Ignition-project storage is acceptable for MVP.

**Proposed direction:** Documented as a side-note in Plan v0.2 §K Phase 0 gates ("Label template scope"). No new OI needed unless MPP pushes back; default to filesystem/Ignition-project templates and revisit if MPP wants in-DB editability.

### Gap 5 — Workstation `AutomationViewCellID` + `NavigationLevelID`

Flexware `Workstation` table (`Flexware_MES_Location_Model.png` + `Flexware_MES_DashboardConfiguration.png`) carries two columns we haven't modelled:

- `AutomationViewCellID` — which Cell this terminal's automation view points at. Used for supervisor terminals that monitor a different Cell than their physical location.
- `NavigationLevelID` — which navigation-level the operator starts at (Cell / Line / Area). Probably a UX-tier cap for role-gated navigation.

**Our coverage:** `DefaultScreen` LocationAttribute + AD-role-based navigation covers the functional need. `AutomationViewCellID` is a legacy workaround for the Flexware pattern of one client serving multiple physical roles; Ignition Perspective's session-bound URL handles this natively.

**Recommendation:** No action. Confirm at Phase 0 that no supervisor setup relied on `AutomationViewCellID`.

## 5. Legacy Patterns NOT Reproduced (By Design) — Verification

This section confirms our deliberate rejections don't hide functional loss.

### 5.1 `*DashboardConfiguration` family (Plan v0.2 §H)

Flexware DB has nine variability tables driving the dashboards: `LotCreateDashboardConfiguration`, `LotTrackingDashboardConfiguration`, `WorkOrderDashboardConfiguration`, `FinalAssemblyDashboardConfiguration`, `CreateAllocationDashboardConfiguration`, `SortDashboardConfiguration`, `MaterialLabelPrintDashboardConfiguration`, `WorkstationDashboardConfiguration` (junction), `MaterialClassesView` (adjacent).

Each one stores columns like `RefreshInterval`, `LotHistoryDisplayCount`, `SaveButtonUserInterfaceScriptID`, `LotTrackingType`, `PopulateFixedQuantityOnItemSelection`, `PopulateInProcessQuantityOnItemSelection`, `PopulateMaterialDefaultBasketQuantity`, `AssemblyLotScanningIsEnabled`.

**Verification that we cover every variability surface:**

| Flexware variability flag | Our equivalent | Status |
|---|---|---|
| `RefreshInterval` (ms) | Ignition Perspective session-auto-refresh + Named Query polling interval (project-level setting) | 🆗 |
| `LotHistoryDisplayCount` | Hard-coded in per-terminal Perspective view (or project-level named constant) | 🆗 |
| `SaveButtonUserInterfaceScriptID` | Explicitly NOT reproduced (Plan v0.2 §I) — Perspective event handlers instead, version-controlled | 🆗 |
| `LotTrackingType` (switch between modes) | Purpose-built views (Machining IN vs Machining OUT vs Trim IN vs …) | 🆗 |
| `LotHistoryAtSourceLocation*` (pull from Area / ProductionLine / WorkCell) | `TrackingMode` LocationAttribute on Cell (Area / Line / Cell) | 🆗 |
| `PopulateFixedQuantityOnItemSelection`, `FixedQuantityToPopulate` | Different views per function — Die Cast view auto-populates from ContainerConfig, Trim view auto-populates from weight | 🆗 |
| `PopulateInProcessQuantityOnItemSelection` | FDS-05-031 derived TotalInProcess — shown in header on all views | 🆗 |
| `PopulateMaterialDefaultBasketQuantity` | OI-18 `Parts.ItemLocation.DefaultQuantity` column (v1.8) | 🆗 |
| `AssemblyLotScanningIsEnabled` | `LocationAttribute` on Cell (`RequiresMaterialVerification` BIT — per FDS-06-011) — operator verify Yes/No | 🆗 |
| `MoveButtonUserInterfaceScriptID` | Perspective event handlers | 🆗 |

**Verdict:** No hidden functional loss. Every configurable surface that mattered maps to either (a) a purpose-built view per function, (b) a LocationAttribute on Cell, or (c) a pre-existing FDS requirement. The pattern we drop is the **runtime-JS-in-DB** config engine, which is both an operational (no version control) and security (stored JS execution) hazard.

### 5.2 `UserInterfaceScript` + `UserInterfaceScriptReference` (Plan v0.2 §I)

Flexware stores JavaScript in a `UserInterfaceScript` table and references it from dashboard save-buttons via `UserInterfaceScriptReference`. This was the "customize behaviour per customer" pattern.

**Our replacement:** All per-terminal / per-function behaviour lives in Ignition Perspective project scripts (version-controlled, deployed with the project). Configuration-side variance lives in stored procedures (Configuration Tool reads + Plant-Floor writes, both auditable).

**Verdict:** No loss. We gain version control, auditability, and the ability to code-review behaviour changes. Confirmed — no legacy screen requires in-DB runtime JS.

### 5.3 Clock number + PIN (Plan v0.2 §A)

Flexware `Workstation.AutoLogonUsername` + `Workstation.AutoLogonPassword` + whatever PIN table we haven't seen drove the legacy shop-floor identity model.

**Our replacement:** Initials-only presence + per-action AD elevation (FDS-04-001..010, Phase C 2026-04-21).

**Verdict:** Legacy model had a classic PIN-screen-drift problem (see "Assume identity of another workstation" — an operator could take over a terminal with another operator's pre-configured AutoLogon). Our model is strictly more traceable. MPP signed off on the rewrite at the 2026-04-20 meeting.

### 5.4 `ProductionOrder` as a separate layer above `WorkOrder`

Flexware has a two-tier `ProductionOrder` → `WorkOrder` model (see `Flexware_MES_OrderDiagram.png`). Our model collapses this into a single `Workorder.WorkOrder` with three types (Demand / Maintenance / Recipe).

**Verification:** Flexware `ProductionOrder` carries `RequiredQuantity`, `AdjustedQuantity`, `SuspendAtQuantity`, `CompletedQuantity`, `RemainingQuantity` — these are all our `WorkOrder` cumulative-quantity fields. The two-tier pattern in Flexware exists because one ProductionOrder can decompose to multiple WorkOrders (different WorkCells, different route steps). In our model, **one `WorkOrder` + many `WorkOrderOperation` rows** achieves the same grain with fewer joins.

**Risk:** If MPP's current process has humans working at the ProductionOrder tier (scheduling person sees POs, not WOs), we need a Supervisor-tier screen that lists "ProductionOrders" computed as `{ItemId, RouteTemplateId, cumulative planned quantity}` rollups over WOs. Phase 0 question.

**Verdict:** Architecturally cleaner. Flag for Phase 0 Ben walk-through.

### 5.5 `Container.TrackingMode` + `WorkCell.TrackingMode` + `WorkCell.RequireCastPartAllocationOverride`

Flexware per-WorkCell flags (see `Flexware_MES_Location_Model.png`, `Flexware_MES_ContainerTracking.png`):

- `WorkCell.TrackingMode TINYINT` — how tightly the Cell tracks material. Probably an enum like `NonSerialized` / `Serialized` / `LotOnly`.
- `WorkCell.RequireCastPartAllocationOverride BIT` — per-Cell supervisor-override requirement for cast parts.
- `Container.TrackingMode TINYINT` — per-container tracking level.
- `Workstation.MaterialAllocationRequired BIT` — per-terminal allocation gate.

**Our coverage:** All four live in our LocationAttribute pattern (FDS-02-005 Polymorphic Kinds). **However**, OI-28 (Work Cell "Require override for cast parts" flag) is still ⬜ open. The other three are implicitly covered but not explicitly documented.

**Recommendation:** Phase 0 Ben walk-through to confirm the enum values for `TrackingMode` (gating OI-08 final resolution). OI-28 needs MPP confirmation.

### 5.6 Role table (in-DB roles) → AD group elevation

Flexware has `RoleAddActiveDirectoryGroupView` — indicating Roles in Flexware are **hybrid**: in-DB roles that reference AD groups. Our model is strict AD-group-as-role (FDS-04-006). The simplification was approved at Phase C (2026-04-21).

**Verdict:** No loss. Verified.
