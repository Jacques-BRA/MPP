location MVP

Plant model following ISA-95, implemented as a polymorphic three-tier classification: LocationType (ISA-95 tier: Enterprise, Site, Area, Work Center, Cell) → LocationTypeDefinition (the kind of location within a tier: Terminal, DieCastMachine, CNCMachine, InventoryLocation, Scale, etc.) → LocationAttributeDefinition (attribute schema per kind). Every Location instance references exactly one LocationTypeDefinition, which determines both its tier and its attribute set. Different definitions within the same tier carry entirely different attribute schemas — a Terminal has IpAddress, a DieCastMachine has Tonnage. Hierarchy is self-referential via ParentLocationId, queried with recursive CTEs. Users authenticate via Active Directory; roles managed in Ignition.

Tables

LocationType

The five ISA-95 tiers: Enterprise (0), Site (1), Area (2), WorkCenter (3), Cell (4). Seeded at deployment, not operator-editable. Answers "what tier of the hierarchy?"

LocationTypeDefinition

The polymorphic kinds within a type. For the Cell tier: Terminal, DieCastMachine, CNCMachine, TrimPress, AssemblyStation, InspectionStation, InventoryLocation, Scale, SortCage. Every Location FKs to exactly one definition. Answers "what kind of thing is this?" Extensible without schema changes. Carries an Icon column for Perspective Tree component display (e.g., material/precision_manufacturing).

LocationAttributeDefinition

Attribute schema per kind. Terminal gets IpAddress, DefaultPrinter, HasBarcodeScanner. DieCastMachine gets Tonnage, NumberOfCavities, RefCycleTimeSec. Different kinds, completely disjoint attribute sets. Each attribute specifies name, data type, required flag, default, UOM, and sort order.

Location

Self-referential hierarchy table. Every node in the plant model is a row — from the Enterprise root down to individual cells. LocationTypeDefinitionId determines both tier and attribute schema via join to LocationType. ParentLocationId forms the adjacency list. Type-specific attributes live in LocationAttribute. Queried with recursive CTEs for ancestor/descendant traversal.

LocationAttribute

Actual attribute values per location, keyed to LocationAttributeDefinition. A Terminal cell might have IpAddress = "10.1.5.42" and DefaultPrinter = "ZEB-DC-05". A DieCastMachine cell might have Tonnage = 350. Integrity rule (enforced in app logic): the attribute definition must belong to the location's own definition.

Terminals in the new model

Terminals are no longer their own LocationType. Terminal is a LocationTypeDefinition under the Cell tier — one kind of Cell among many. A DieCastMachine is another kind of Cell. Both are Cell-tier locations but carry completely different attribute schemas. Event tables carry TerminalLocationId (where operator stands) and LocationId (which machine they scanned).

AppUser MVP v1.6

Realigned to the initials-based security model (OI-06, Phase C of the 2026-04-20 OI refactor). Initials is now NOT NULL UNIQUE — every user has an initial stamp used as the shop-floor event attribution. AdAccount is nullable (filtered UNIQUE where NOT NULL) so operator-class rows can exist without an AD identity. A CHECK (IgnitionRole IS NULL OR AdAccount IS NOT NULL) constraint binds elevated Ignition roles to AD-backed accounts. ClockNumber and PinHash are flagged legacy and will be dropped in the Phase G migration.

parts MVP

Item master data, bills of material, manufacturing routes, and operation templates. Defines what MPP makes, what goes into it, and how it moves through the plant. BOMs and routes are versioned with effective dates and soft deletes so production records always point to the spec active at time of manufacture. Container configurations define Honda-required packing rules per product.

Tables

Item MVP v1.9c

The part/product master. Every raw material, component, sub-assembly, and finished good. Carries MPP part number, optional Macola ERP number, DefaultSubLotQty, MaxLotSize (repurposed v1.9 as PartsPerBasket), MaxParts (v1.9c, OI-12 correction — per-Item per-Location hard cap), unit weight, and (v1.8) CountryOfOrigin (ISO 3166-1 alpha-2, OI-19).

ItemType

Classifies items: Raw Material, Component, Sub-Assembly, Finished Good, Pass-Through. Drives UI behavior and routing logic.

Uom

Unit of measure reference (EA, LB, KG, etc.). First-class column on every quantitative field per best practices.

Bom / BomLine

Versioned bill of materials. Header links parent item to component structure with EffectiveFrom / DeprecatedAt. Lines specify child items and quantities per parent.

RouteTemplate / RouteStep

Versioned manufacturing routes. Each template defines the ordered sequence of operations for a product. Steps reference operation templates and carry sequence numbers.

OperationTemplate

Defines what data to collect at a given operation type: die info, cavity, weight, counts, material verification, serial numbers. Linked to an area.

ItemLocation MVP v1.9d

Part-to-location eligibility. v1.9d (OI-18 extension): LocationId accepts any Location tier — Area / WorkCenter / Cell. Eligibility at a specific Cell is resolved by walking UP the Location hierarchy from the scanned Cell to Site; first matching ItemLocation row returns eligible. Engineering declares the coarsest appropriate tier (one "Part 5G0 eligible across Die Cast Area" row replaces per-Cell enumeration). Helper proc Parts.ItemLocation_IsEligible(@ItemId, @CellLocationId) encapsulates the walk. v1.8 consumption metadata (MinQuantity / MaxQuantity / DefaultQuantity / IsConsumptionPoint) lives on the matching row at whatever tier — ancestor-row metadata applies when no Cell-specific override exists.

ContainerConfig

Honda-specified packing rules per product: trays per container, parts per tray, serialized flag, returnable dunnage code, customer code. ClosureMethod / TargetWeight (OI-02, confirmed 2026-04-24) drive weight-based closure on non-serialized lines. Note: the v1.8 MaxParts column was moved from ContainerConfig to Parts.Item in v1.9c (OI-12 correction) — MaxParts is a Part attribute, not a container attribute.

lots MVP

The core of the MES. Tracks every LOT from creation through its entire lifecycle: movements between locations, splits into sub-LOTs, merges, consumption into finished goods, and final shipping. Includes the genealogy graph (adjacency list with recursive CTE support), containers, serialized parts, and shipping labels. All state changes are recorded as immutable history records for 20-year traceability.

Tables

LotOriginType

How a LOT entered the system: Manufactured (created at die cast), Received (pass-through parts received on-site), ReceivedOffsite (pass-through parts received at a remote warehouse). Enables filtering and reporting by origin path.

Lot MVP v1.9

The central entity. Unique LotName (minted from Lots.IdentifierSequence, Code=Lot, format MESL{0:D7}), links to item and origin type, carries piece count, weight, vendor LOT reference, quality status, current location. ParentLotId is the adjacency-list link for Machining sub-LOTs only (cavity-parallel LOTs at Die Cast are peers, not sublots — OI-09 closed).

v1.9 additions: ToolId BIGINT NULL FK → Tools.Tool and ToolCavityId BIGINT NULL FK → Tools.ToolCavity — system of record for die-cast-origin LOTs (validated against active ToolAssignment on the Cell at Lot_Create; cavity must belong to tool and be Active). NULL for non-die-cast origins. NULL after Lot_Merge on blended-origin LOTs. Downstream LOTs do NOT carry — Honda-trace via LotGenealogy walk. Pre-v1.9 DieNumber / CavityNumber NVARCHAR columns are legacy (retained for cutover; scheduled for removal).

IdentifierSequence MVP v1.9

Replaces Flexware's IdentifierFormat (OI-31). Drives all MPP-internal identifier minting via Lots.IdentifierSequence_Next @Code: Lot LTT barcodes (MESL{0:D7}) and SerializedItem IDs (MESI{0:D7}). Honda AIM shipper IDs are out of scope — those come from AIM.GetNextNumber. Cutover migration seeds LastValue at or above Flexware's live counter values. EndingValue defaults to 9,999,999; the proc raises on rollover without an explicit reset policy.

LotStatusCode

LOT quality states: Good, Hold, Scrap, Closed. BlocksProduction drives interlock logic.

LotGenealogy

Edge table for the genealogy graph. Records every split, merge, and consumption relationship. Supports recursive CTE traversal in both directions.

LotStatusHistory / LotMovement / LotAttributeChange

Immutable append-only logs for status transitions, location changes, and attribute modifications. Every record captures who, when, and which terminal location.

LotLabel

Tracks every LTT barcode label printed or reprinted by the MES. Print reasons: Initial (only for Received / ReceivedOffsite LOTs — Die Cast LTTs are pre-printed outside the MES per FDS-05-020), ReprintDamaged, Split, Merge, SortCageReIdentify. Each record captures the print reason, ZPL content, printer, and operator.

Container / ContainerTray

Shipping containers with trays. Tracks status, AIM shipper ID, hold state, and current location. Container config comes from the parts schema.

SerializedPart / ContainerSerial

Individual laser-etched serial numbers linked to source LOTs. Junction table maps serials to container tray positions. Critical for Sort Cage re-inspection.

ShippingLabel

Every label printed for a container: AIM shipper ID, ZPL content, print time, void status. Labels can be voided and reprinted during Sort Cage workflow.

PauseEvent MVP v1.9g

Append-only place + close lifecycle for operator-driven LOT pauses at a workstation (OI-21 / FDS-05-038). Mirrors Quality.HoldEvent's shape. (LotId, LocationId) with optional PausedReason; nullable ResumedAt + ResumedByUserId + ResumedRemarks close the pause. Filtered UNIQUE on (LotId, LocationId) WHERE ResumedAt IS NULL blocks duplicate open pauses for the same LOT-at-Cell, but the same LOT MAY be open-paused at multiple Cells simultaneously (Machining + Assembly partial-progress). Pause is orthogonal to WorkOrderStatus, OperationStatus, and LotStatusCode — no Paused row added to any of those code tables; pause does NOT write a DowntimeEvent. No TTL — paused LOTs persist across shifts and operators (resume MAY be by a different operator). Drives the Paused-LOT indicator (count + tap-through detail list) on every workstation screen.

AimShipperIdPool MVP v1.9h

Local FIFO pool of pre-fetched Honda AIM Shipper IDs (UJ-04 / FDS-07-010). Each row's lifecycle is binary: available (ConsumedAt IS NULL) or consumed (terminal — never returns to available, even on container void / re-pack). A background Gateway timer script (~30s cadence) calls AIM.GetNextNumber to keep the pool topped up toward AimPoolConfig.TargetBufferDepth; Container_Complete claims one row via UPDATE TOP (1) WITH (UPDLOCK, READPAST, ROWLOCK) ... OUTPUT ... ORDER BY FetchedAt — sub-millisecond, never blocks on AIM. FetchedInterfaceLogId FK → Audit.InterfaceLog gives end-to-end provenance from a Container.AimShipperId back to the originating AIM call. Empty pool = hard fail (FDS-07-010a): close transaction rolls back; line stops until pool refills.

AimPoolConfig MVP v1.9h

Single-row config for the AIM Shipper ID pool (FDS-07-010c). Four operator-tunable thresholds via Configuration Tool (Admin-elevated): TargetBufferDepth (default 50), TopupThreshold (default 30), AlarmWarningDepth (default 20), AlarmCriticalDepth (default 10). CHECK constraint enforces strict ordering Critical < Warning < Topup < Target. Two-tier alarm: wallboard tile turns Warning at depth < AlarmWarningDepth; supervisor alarm + IT notification at depth < AlarmCriticalDepth; severity upgrades to "POOL EXHAUSTED" at depth = 0. Single-row enforced by CHECK (Id = 1).

workorder Mixed Scope

Internal work order context that operators never see. Auto-generated when production activity begins, providing bookkeeping for production events, material consumption, and reject tracking. Links LOTs to machines, routes, and operators. Every good count, bad count, and material consumption is recorded as an immutable event.

Work order tables (WorkOrder, WorkOrderOperation, WorkOrderType) are MVP-LITE per OI-07 — auto-generated and populated, but invisible to operators (no WO screens). Production/consumption/reject event tables are MVP. Events have nullable FKs to WorkOrderOperation, so they function independently. v1.9b (2026-04-24 OI-07 correction): MVP ships with one active WorkOrderType row — Production — preserving the MVP-LITE bookkeeping. Demand (planned preventative maintenance) and Maintenance (emergency) are genuinely separate future WO types under MPP's taxonomy but are NOT built in this project; the code table stays as a future hook so a separate maintenance-engine project can INSERT those rows without schema change. The nullable ToolId FK on WorkOrder remains as a schema hook for the future Maintenance flow. The pre-correction three-type model (Demand / Maintenance / Recipe) and the Recipe concept have been retired.

Tables

WorkOrderType MVP-Lite v1.9b

Read-only code table. Seed corrected v1.9b (2026-04-24 OI-07): single row Production — the existing MVP-LITE bookkeeping flow (auto-generated on LOT start, invisible to operators). Added v1.7 as part of the Phase B refactor; originally seeded with three rows (Demand / Maintenance / Recipe) which were retired when Jacques clarified the 2026-04-20 meeting note had been mis-recorded. Future additions — Demand (planned preventative maintenance) and Maintenance (emergency) — are out of scope for this project; the code-table mechanism is the hook so a future maintenance-engine project can INSERT those rows without schema change. Existing WorkOrder rows backfill to Production.

WorkOrder MVP-Lite

Auto-generated internal work order. Links an item to a route template. Operators never interact with this — it provides context for aggregating production data. (OI-07: MVP-Lite, included but hidden, no WO screens.) v1.7 adds WorkOrderTypeId (NOT NULL DEFAULT Production-Id per v1.9b correction) and nullable ToolId FK → Tools.Tool. ToolId is a future-Maintenance schema hook; not populated or enforced in MVP.

WorkOrderOperation MVP-Lite

Individual operation execution within a work order. Links a route step to a specific machine, tracks start/completion times and operator.

ProductionEvent MVP v1.9 (reshaped)

Checkpoint-shape event table (v1.9). Per FRS §2.1.2 operator-driven capture: each row is one checkpoint — a discrete moment when an operator logs something (basket close, area handoff, complete + move, scrap action, quality transition). Operators are not at the terminal per shot.

Header carries LotId, OperationTemplateId, EventAt, cumulative ShotCount/ScrapCount, optional ScrapSourceId (populated only for scrap-driving checkpoints, OI-20), optional WeightValue/WeightUomId, AppUserId, TerminalLocationId. Deltas between checkpoints are derived by the reader via LAG() window over (LotId, EventAt). Required index (LotId, EventAt DESC).

Deliberately NOT on this table (v1.9 reshape): no LocationId (derive from LotMovement), no ItemId (derive from Lot.ItemId), no DieIdentifier/CavityNumber (derive from Lot.ToolId/Lot.ToolCavityId), no per-event GoodCount/NoGoodCount (replaced by cumulative ShotCount/ScrapCount with LAG()-derived deltas), no StartedAt/EndedAt (interval start is the previous event for the same LOT).

Additional DataCollectionField values configured on the template land in ProductionEventValue children.

ScrapSource MVP v1.8

Read-only code table distinguishing the two scrap paths: Inventory (scrap from unallocated LOT stock, no workstation context) and Location (scrap from in-process material at a specific Cell). Mirrors the legacy Flexware "Scrap from inventory" vs "Scrap from the selected location" buttons on the Lot Details screen (OI-20).

ProductionEventValue MVP

Child of ProductionEvent. Holds any DataCollectionField value that isn't promoted to a hot column on the event. One row per (event, field), with both a string Value and a typed NumericValue/UomId where applicable. Enables extending the data-collection vocabulary without schema changes.

ConsumptionEvent MVP

Records which source LOT supplied parts and which output LOT or container was produced. The backbone of genealogy across the assembly boundary.

RejectEvent MVP

Detailed reject/scrap records linked to production events. Defect code, quantity, charge-to area, operator remarks. Feeds quality analysis and OEE quality-rate.

Downtime tracking, shift management, and materialized OEE metrics. ~660 reason codes across areas. tools Mixed Scope

Added v1.7 (Phase B of the 2026-04-20 OI review refactor). Promotes Tool from a LocationAttribute historical-snapshot pattern to a first-class polymorphic subsystem covering dies, cutters, jigs, gauges, assembly fixtures, and trim tools. Mirrors the Location pattern but collapsed to a single header table: ToolType is a grouping (not a hierarchy), ToolAttributeDefinition is the per-type attribute schema, Tool is the concrete row. ToolCavity children exist only for tool types with HasCavities = 1 (currently only dies). ToolAssignment is the append-only check-in/out history against Cells. DieRank + DieRankCompatibility back the OI-05 cross-die merge rule.

All ten tables are MVP. Configuration Tool CRUD screens for all entities are MVP (Phase 9). Maintenance WO flow is FUTURE — schema hook (Workorder.WorkOrderType = Maintenance + nullable Workorder.WorkOrder.ToolId) is MVP. Tool-life threshold alarms are FUTURE (scheduled Gateway Script pattern when MPP asks; shot counts derive from Workorder.ProductionEvent). Cross-plant tool transfer is FUTURE.

Tables

ToolType MVP

Polymorphic kinds (seeded read-only): Die, Cutter, Jig, Gauge, AssemblyFixture, TrimTool. Carries HasCavities BIT — gates whether ToolCavity children are valid for tools of this type. Icon column for Perspective tree display.

ToolAttributeDefinition MVP

Per-type attribute schema. Mirrors Location.LocationAttributeDefinition. Seeded empty — engineering populates via the Config Tool as real tools arrive (e.g., CycleTimeSec and Tonnage for dies, InsertCount for cutters).

Tool MVP

System of record for tool identity. Code, name, description, ToolTypeId, StatusCodeId, and nullable DieRankId (Die-type only — application-level validation, no CHECK). No shot counter: shot counts derive from Workorder.ProductionEvent group-by to avoid drift. DeprecatedAt soft-delete is separate from the business state StatusCode = Retired.

ToolAttribute MVP

Attribute values, keyed by (ToolId, ToolAttributeDefinitionId). Stored as NVARCHAR; interpreted per the definition's DataType. Mirrors Location.LocationAttribute.

ToolCavity MVP

Child of Tool. Valid only for types with HasCavities = 1. Three-state status code (Active / Closed / Scrapped) tracks whether the cavity is producing, shut off, or physically destroyed. Append-only check-in/out history against Cells. Filtered UNIQUE on ToolId WHERE ReleasedAt IS NULL enforces one active mount per tool. Mount/release requires the FDS-04-007 AD elevation prompt — it's on the supervisor-gated action list.

ToolStatusCode · ToolCavityStatusCode MVP

Read-only code tables. ToolStatusCode: Active / UnderRepair / Scrapped / Retired. ToolCavityStatusCode: Active / Closed / Scrapped. Both seeded at migration time.

DieRank MVP

Code table for die-quality ranks (e.g., A–E). Ships empty — MPP Quality owes the authoritative list. Config Tool has a Die Rank admin screen for engineering to populate once received.

DieRankCompatibility MVP

Junction: (RankAId, RankBId, CanMix). Pairs stored canonicalised (smaller Id first). Ships empty. audit MVP

Immutable, append-only logging. Four log streams: OperationLog (shop-floor actions), ConfigLog (admin changes that succeeded), InterfaceLog (AIM/PLC/ERP communications), and FailureLog (attempted-but-rejected operations). 20-year retention. BIGINT PKs for high-volume append. All writes flow through four shared stored procedures (Audit_LogOperation, Audit_LogConfigChange, Audit_LogInterfaceCall, Audit_LogFailure) — entity-specific procs never write to log tables directly.

Tables

LogSeverity

Info, Warning, Error, Critical.

LogEventType

Normalized vocabulary for what happened: Created, Updated, Deprecated, LotCreated, LotMoved, LotSplit, LotStatusChanged, ProductionRecorded, ConsumptionRecorded, HoldPlaced, HoldReleased, LabelPrinted, ContainerCompleted, UserLogin, etc. Shared across OperationLog, ConfigLog, and FailureLog. Prevents free-text drift.

LogEntityType

Normalized vocabulary for what was affected: Lot, Container, SerializedPart, WorkOrder, Item, Location, LocationTypeDefinition, AppUser, Bom, RouteTemplate, OperationTemplate, QualitySpec, DefectCode, DowntimeReasonCode, HoldEvent, etc. Shared across all four log tables.

OperationLog

Every successful shop-floor action: user, terminal location, machine location, affected entity, old/new values. Written by Audit_LogOperation inside the mutating proc's transaction — atomic with the data. Primary regulatory audit trail.

ConfigLog

Every successful engineering/admin configuration change: plant model edits, item master updates, route modifications, BOM edits, quality spec versions. Written by Audit_LogConfigChange inside the transaction — atomic with the data.

InterfaceLog

External system communications: AIM EDI, PLC/OPC, future Macola/Intelex. Direction, payloads, errors. Written by Audit_LogInterfaceCall. No user FK — system-initiated. High-fidelity mode captures full request/response payloads per FRS 3.17.4.

FailureLog

Attempted-but-rejected stored procedure calls: parameter failures, business-rule violations ("can't deprecate with active dependents"), FK mismatches, and caught exceptions. Written by Audit_LogFailure from every validation-failure path and every CATCH handler. Writes happen outside the rolled-back transaction so the failure record survives. Drives the Configuration Tool's Failure Log Browser (top rejection reasons, top failing procs). Complements ConfigLog/OperationLog which record what succeeded.
