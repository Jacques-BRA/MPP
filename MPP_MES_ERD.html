location MVP

Plant model following ISA-95, implemented as a polymorphic three-tier classification: LocationType (ISA-95 tier: Enterprise, Site, Area, Work Center, Cell) → LocationTypeDefinition (the kind of location within a tier: Terminal, DieCastMachine, CNCMachine, InventoryLocation, Scale, etc.) → LocationAttributeDefinition (attribute schema per kind). Every Location instance references exactly one LocationTypeDefinition, which determines both its tier and its attribute set. Different definitions within the same tier carry entirely different attribute schemas — a Terminal has IpAddress, a DieCastMachine has Tonnage. Hierarchy is self-referential via ParentLocationId, queried with recursive CTEs. Users authenticate via Active Directory; roles managed in Ignition.

Tables

LocationType

The five ISA-95 tiers: Enterprise (0), Site (1), Area (2), WorkCenter (3), Cell (4). Seeded at deployment, not operator-editable. Answers "what tier of the hierarchy?"

LocationTypeDefinition

The polymorphic kinds within a type. For the Cell tier: Terminal, DieCastMachine, CNCMachine, TrimPress, AssemblyStation, InspectionStation, InventoryLocation, Scale, SortCage. Every Location FKs to exactly one definition. Answers "what kind of thing is this?" Extensible without schema changes. Carries an Icon column for Perspective Tree component display (e.g., material/precision_manufacturing).

LocationAttributeDefinition

Attribute schema per kind. Terminal gets IpAddress, DefaultPrinter, HasBarcodeScanner. DieCastMachine gets Tonnage, NumberOfCavities, RefCycleTimeSec. Different kinds, completely disjoint attribute sets. Each attribute specifies name, data type, required flag, default, UOM, and sort order.

Location

Self-referential hierarchy table. Every node in the plant model is a row — from the Enterprise root down to individual cells. LocationTypeDefinitionId determines both tier and attribute schema via join to LocationType. ParentLocationId forms the adjacency list. Type-specific attributes live in LocationAttribute. Queried with recursive CTEs for ancestor/descendant traversal.

LocationAttribute

Actual attribute values per location, keyed to LocationAttributeDefinition. A Terminal cell might have IpAddress = "10.1.5.42" and DefaultPrinter = "ZEB-DC-05". A DieCastMachine cell might have Tonnage = 350. Integrity rule (enforced in app logic): the attribute definition must belong to the location's own definition.

Terminals in the new model

Terminals are no longer their own LocationType. Terminal is a LocationTypeDefinition under the Cell tier — one kind of Cell among many. A DieCastMachine is another kind of Cell. Both are Cell-tier locations but carry completely different attribute schemas. Event tables carry TerminalLocationId (where operator stands) and LocationId (which machine they scanned).

AppUser

MES users backed by Active Directory. The AdAccount is the authentication identity. Clock number and PIN hash support shop-floor identification at terminals. The IgnitionRole field references the role name configured in Ignition's internal identity provider.

parts MVP

Item master data, bills of material, manufacturing routes, and operation templates. Defines what MPP makes, what goes into it, and how it moves through the plant. BOMs and routes are versioned with effective dates and soft deletes so production records always point to the spec active at time of manufacture. Container configurations define Honda-required packing rules per product.

Tables

Item

The part/product master. Every raw material, component, sub-assembly, and finished good. Carries MPP part number, optional Macola ERP number, default sub-lot quantity, max lot size, and unit weight.

ItemType

Classifies items: Raw Material, Component, Sub-Assembly, Finished Good, Pass-Through. Drives UI behavior and routing logic.

Uom

Unit of measure reference (EA, LB, KG, etc.). First-class column on every quantitative field per best practices.

Bom / BomLine

Versioned bill of materials. Header links parent item to component structure with EffectiveFrom / DeprecatedAt. Lines specify child items and quantities per parent.

RouteTemplate / RouteStep

Versioned manufacturing routes. Each template defines the ordered sequence of operations for a product. Steps reference operation templates and carry sequence numbers.

OperationTemplate

Defines what data to collect at a given operation type: die info, cavity, weight, counts, material verification, serial numbers. Linked to an area.

ItemLocation

Part-to-machine eligibility mapping. Certain parts can only run on certain machines. MES uses this to prevent operators from entering LOTs on incompatible equipment. References the location table.

ContainerConfig

Honda-specified packing rules per product: trays per container, parts per tray, serialized flag, returnable dunnage code, customer code.

lots MVP

The core of the MES. Tracks every LOT from creation through its entire lifecycle: movements between locations, splits into sub-LOTs, merges, consumption into finished goods, and final shipping. Includes the genealogy graph (adjacency list with recursive CTE support), containers, serialized parts, and shipping labels. All state changes are recorded as immutable history records for 20-year traceability.

Tables

LotOriginType

How a LOT entered the system: Manufactured (created at die cast), Received (pass-through parts received on-site), ReceivedOffsite (pass-through parts received at a remote warehouse). Enables filtering and reporting by origin path.

Lot

The central entity. Unique lot name (LTT barcode), links to item and origin type, carries piece count, weight, die/cavity info, vendor LOT reference, quality status, and current location. ParentLotId provides the adjacency-list link for sub-LOTs.

LotStatusCode

LOT quality states: Good, Hold, Scrap, Closed. BlocksProduction drives interlock logic.

LotGenealogy

Edge table for the genealogy graph. Records every split, merge, and consumption relationship. Supports recursive CTE traversal in both directions.

LotStatusHistory / LotMovement / LotAttributeChange

Immutable append-only logs for status transitions, location changes, and attribute modifications. Every record captures who, when, and which terminal location.

LotLabel

Tracks every LTT barcode label printed or reprinted by the MES. Print reasons: Initial (only for Received / ReceivedOffsite LOTs — Die Cast LTTs are pre-printed outside the MES per FDS-05-020), ReprintDamaged, Split, Merge, SortCageReIdentify. Each record captures the print reason, ZPL content, printer, and operator.

Container / ContainerTray

Shipping containers with trays. Tracks status, AIM shipper ID, hold state, and current location. Container config comes from the parts schema.

SerializedPart / ContainerSerial

Individual laser-etched serial numbers linked to source LOTs. Junction table maps serials to container tray positions. Critical for Sort Cage re-inspection.

ShippingLabel

Every label printed for a container: AIM shipper ID, ZPL content, print time, void status. Labels can be voided and reprinted during Sort Cage workflow.

workorder Mixed Scope

Internal work order context that operators never see. Auto-generated when production activity begins, providing bookkeeping for production events, material consumption, and reject tracking. Links LOTs to machines, routes, and operators. Every good count, bad count, and material consumption is recorded as an immutable event.

Work order tables (WorkOrder, WorkOrderOperation) are MVP-LITE per OI-07 — auto-generated and populated, but invisible to operators (no WO screens). Production/consumption/reject event tables are MVP. Events have nullable FKs to WorkOrderOperation, so they function independently.

Tables

WorkOrder MVP-Lite

Auto-generated internal work order. Links an item to a route template. Operators never interact with this — it provides context for aggregating production data. (OI-07 resolved to MVP-Lite: included but hidden, no WO screens.)

WorkOrderOperation MVP-Lite

Individual operation execution within a work order. Links a route step to a specific machine, tracks start/completion times and operator.

ProductionEvent MVP

Immutable record of production activity: good count, no-good count, operator, terminal location, timestamp against a LOT and machine. Die/cavity info lives on the Lot record (one LOT = one die/cavity), not on the event.

ConsumptionEvent MVP

Records which source LOT supplied parts and which output LOT or container was produced. The backbone of genealogy across the assembly boundary.

RejectEvent MVP

Detailed reject/scrap records linked to production events. Defect code, quantity, charge-to area, operator remarks. Feeds quality analysis and OEE quality-rate.

quality Mixed Scope

Specification-driven inspection, sample tracking, non-conformance records, and hold management. Quality specs are versioned so production records reference the spec active at time of inspection. Hold events integrate with LOT status transitions to enforce manufacturing interlocks. Supports file attachments for test result documentation.

Holds and inspections are MVP. Sampling workflows are CONDITIONAL. NCM/Failure Analysis is FUTURE — the NonConformance table is retained to preserve the hold↔NCM design separation and will activate without schema changes.

Tables

DefectCode MVP

~170 reject/defect reason codes by area. IsExcused flag affects OEE quality calculations. Supports both reject tracking (MVP) and future NCM.

QualitySpec / QualitySpecVersion / QualitySpecAttribute MVP

Specification-driven quality schema. Specs define what to inspect per item/operation. Versions ensure historical accuracy. Attributes define measurable values with targets and limits.

QualitySample / QualityResult Conditional

Inspection events against LOTs. Each sample records overall result; individual results capture measured values against spec attributes. MVP for basic inspections; expanded sampling workflows are conditional.

NonConformance Future

Formal NC record against a LOT: defect code, quantity, disposition, resolution tracking. Not in current scope — retained for future NCM/Failure Analysis activation. Completes the hold↔NCM design separation.

HoldEvent MVP-Expanded

Records when a LOT is placed on quality hold (type, reason, user) and when the hold is released. Same lifecycle pattern as DowntimeEvent — created on placement, updated on release. Active holds have ReleasedAt IS NULL. Drives LOT status transitions to/from HOLD. Expanded beyond legacy capabilities.

QualityAttachment MVP

File uploads (CSV, XLSX, PDF, images) linked to samples or non-conformances.

oee Mixed Scope

Downtime tracking, shift management, and materialized OEE metrics. ~660 reason codes across areas. OEE snapshots are calculated per machine per shift as a materialized reporting layer — not computed live. The normalized core is the system of record; the snapshot is a derivative following OLTP/OLAP separation.

Downtime tracking and shifts are MVP. OEE snapshots and KPI dashboards are FUTURE — the oee_snapshot table is retained because it is purely derivative of MVP data. Activation requires only a scheduled calculation job, no new data capture.

Tables

DowntimeReasonType MVP

Top-level categorization: Equipment, Miscellaneous, Mold, Quality, Setup, Unscheduled.

DowntimeReasonCode MVP

~660 specific reasons by area and type. IsExcused determines OEE availability impact.

ShiftSchedule / Shift MVP

Named shift patterns and actual shift instances. Time boundaries for downtime context and production reporting.

DowntimeEvent MVP

Append-only. Machine, reason code, start/end, source (manual/PLC), operator. Open events have null EndedAt. ShotCount captures warm-up/setup shots when ReasonType = Setup (per UJ-14).

OeeSnapshot Future

Materialized Availability × Performance × Quality per machine per shift. Recalculated by scheduled job, never the system of record. Not in current scope — retained as a natural derivative of MVP downtime and production data.

audit MVP

Immutable, append-only logging. Four log streams: OperationLog (shop-floor actions), ConfigLog (admin changes that succeeded), InterfaceLog (AIM/PLC/ERP communications), and FailureLog (attempted-but-rejected operations). 20-year retention. BIGINT PKs for high-volume append. All writes flow through four shared stored procedures (Audit_LogOperation, Audit_LogConfigChange, Audit_LogInterfaceCall, Audit_LogFailure) — entity-specific procs never write to log tables directly.

Tables

LogSeverity

Info, Warning, Error, Critical.

LogEventType

Normalized vocabulary for what happened: Created, Updated, Deprecated, LotCreated, LotMoved, LotSplit, LotStatusChanged, ProductionRecorded, ConsumptionRecorded, HoldPlaced, HoldReleased, LabelPrinted, ContainerCompleted, UserLogin, etc. Shared across OperationLog, ConfigLog, and FailureLog. Prevents free-text drift.

LogEntityType

Normalized vocabulary for what was affected: Lot, Container, SerializedPart, WorkOrder, Item, Location, LocationTypeDefinition, AppUser, Bom, RouteTemplate, OperationTemplate, QualitySpec, DefectCode, DowntimeReasonCode, HoldEvent, etc. Shared across all four log tables.

OperationLog

Every successful shop-floor action: user, terminal location, machine location, affected entity, old/new values. Written by Audit_LogOperation inside the mutating proc's transaction — atomic with the data. Primary regulatory audit trail.

ConfigLog

Every successful engineering/admin configuration change: plant model edits, item master updates, route modifications, BOM edits, quality spec versions. Written by Audit_LogConfigChange inside the transaction — atomic with the data.

InterfaceLog

External system communications: AIM EDI, PLC/OPC, future Macola/Intelex. Direction, payloads, errors. Written by Audit_LogInterfaceCall. No user FK — system-initiated. High-fidelity mode captures full request/response payloads per FRS 3.17.4.

FailureLog

Attempted-but-rejected stored procedure calls: parameter failures, business-rule violations ("can't deprecate with active dependents"), FK mismatches, and caught exceptions. Written by Audit_LogFailure from every validation-failure path and every CATCH handler. Writes happen outside the rolled-back transaction so the failure record survives. Drives the Configuration Tool's Failure Log Browser (top rejection reasons, top failing procs). Complements ConfigLog/OperationLog which record what succeeded.
